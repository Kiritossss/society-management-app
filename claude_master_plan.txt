## ⚙️ One-Time Setup (Do This Before Testing Anything)

### Step 1 — Start PostgreSQL
```bash
brew install postgresql@16
brew services start postgresql@16
createdb society_db
```

### Step 2 — Create the .env file
File location: `backend/.env`
Copy from `backend/.env.example` and fill in:
```
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/society_db
SECRET_KEY=<run: openssl rand -hex 32>
DEBUG=True
ACCESS_TOKEN_EXPIRE_MINUTES=30
RATE_LIMIT_PER_MINUTE=60
```

### Step 3 — Run database migrations
```bash
cd /Users/masum/Development/Society/backend
alembic upgrade head
```

### Step 4 — Start the backend server
```bash
cd /Users/masum/Development/Society/backend
uvicorn app.main:app --reload --port 8000
```
API docs available at: http://localhost:8000/docs

## 🧪 How to Create a Society & Test the Full Login Flow

> Do all of this via Swagger UI at http://localhost:8000/docs

### Step 1 — Register a Society
- Open: `POST /api/v1/auth/society/register`
- Click "Try it out" → fill in the body:
```json
{
  "name": "Green Valley Society",
  "address": "123 Main Street, City",
  "contact_email": "admin@greenvalley.com"
}
```
- Click Execute.
- **Copy the `id` field from the response** — this is your `society_id` (UUID).
- Example: `"id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"`

### Step 2 — Register a User (Admin)
- Open: `POST /api/v1/auth/register`
- Add query param `society_id` = the UUID you copied above
- Body:
```json
{
  "full_name": "Your Name",
  "email": "you@example.com",
  "password": "yourpassword123",
  "role": "admin"
}
```
- Click Execute. User is created.

### Step 3 — Login
- Open: `POST /api/v1/auth/login`
- Add query param `society_id` = the UUID
- Body:
```json
{
  "email": "you@example.com",
  "password": "yourpassword123"
}
```
- Copy the `access_token` from the response.

### Step 4 — Authorise in Swagger (to test protected endpoints)
- Click the green **"Authorize"** button at the top of the Swagger page
- Paste your token in the `Bearer` field → click Authorize
- Now all protected endpoints (complaints, etc.) will work

### Step 5 — Login from the Flutter App
- Run the Flutter app: `cd frontend && flutter run -d chrome --web-port 3000`
- Enter the `society_id` UUID, your email, and password on the Login screen
- You will land on the Dashboard and can navigate to Complaints

## Phase 1: Database & Core Setup (Backend)
**Objective**: Establish the foundation of the FastAPI backend and database.
1. Write the SQLAlchemy Base setup (`backend/app/db/base.py`, `backend/app/db/session.py`).
2. Write the database models for `Society` and `User` (`backend/app/models/society.py`, `backend/app/models/user.py`). 
    - Ensure strict multi-tenancy rules (`society_id` on Users).
3. Set up Alembic and generate the first migration schema.

## Phase 2: Authentication & Multi-Tenancy (Backend)
**Objective**: Build secure login and role-based access.
1. Write Pydantic schemas for `User` and `Society` creation & login (`backend/app/schemas/`).
2. Write the authentication dependency (`backend/app/api/dependencies.py`) to verify JWTs, user roles, and extract `society_id` securely.
3. Create the auth endpoints (`backend/app/api/v1/endpoints/auth.py`).

## Phase 3: Flutter App Foundation (Frontend)
**Objective**: Setup the mobile app shell.
1. Configure `pubspec.yaml` with required packages (Riverpod/Provider, Dio/HTTP, GoRouter).
2. Create the core theme and constants (`frontend/lib/core/`).
3. Set up the API networking client (Dio) with an interceptor to automatically attach JWT tokens.
4. Build the Login & Registration screens (`frontend/lib/features/auth/`).

## Phase 4: Core Feature 1 - Complaint Management
*Note: We develop features full-stack, module by module.*
1. **Backend**: 
    - Create `Complaint` model, schemas, and CRUD operations.
    - Build FastAPI endpoints (`backend/app/api/v1/endpoints/complaints.py`).
2. **Frontend**: 
    - Build Flutter UI to list complaints.
    - Build Flutter UI to create a new complaint (with image upload snippet).
    - Connect Flutter UI to the backend endpoints.

## Phase 5: Core Feature 2 - Visitor & Security Management
1. **Backend**: 
    - Create `VisitorLog` model, schemas, and CRUD.
    - Build endpoints ensuring only `Support Staff` (watchmen) can create entries.
2. **Frontend**:
    - Build Watchman Dashboard UI to log entry/exit.
    - Build Resident UI to approve/deny entry.

## Phase 6: Core Feature 3 - Maintenance & Payments
1. **Backend**:
    - Create `Bill` and `Payment` models.
    - Build billing endpoints to auto-generate society dues.
2. **Frontend**:
    - Build Billing Dashboard UI for residents to view dues.
    - Integrate payment gateway (e.g., Stripe) stub on Flutter.

## Phase 7: Polish & Push Notifications
1. **Backend**: Setup Firebase Admin SDK in FastAPI to send FCM push messages on events (e.g., new visitor, new complaint).
2. **Frontend**: Integrate Firebase Cloud Messaging package in Flutter to receive pushes and navigate to the relevant screen.
