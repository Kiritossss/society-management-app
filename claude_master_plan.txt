## Architecture Overview

This is a multi-tenant SaaS product sold to housing societies. Each society gets a 5-letter code (e.g. `MKQWZ`).

### Two Frontends, One Backend

| Frontend | Users | Purpose |
|----------|-------|---------|
| **Admin Web Portal** (React/Next.js) | Admin, Committee | Society setup, unit management, member invites, complaint oversight, billing config, reports |
| **Mobile App** (Flutter) | Residents, Support Staff | Activate account via invite, file complaints, approve visitors, pay bills, book amenities |

### Auth Flow (Invite-Based)

```
1. Admin registers society (web portal or Swagger) → gets 5-letter code
2. Admin logs into web portal using society code + email + password
3. Admin adds members → system generates a unique invite token per member
4. Resident receives invite (SMS/WhatsApp/email link)
5. Resident opens mobile app → enters invite token → sets password → account activated
6. Resident logs in with email + password (society code auto-embedded in their account)
```

**Key principle:** Residents never need to know the society code. Admin controls who joins.

### Multi-Society Users

A person can own flats in multiple societies (e.g. `john@gmail.com` in both Green Valley and Palm Heights). The DB supports this — `UNIQUE(society_id, email)` allows the same email in different societies.

**Mobile login flow for multi-society users:**
```
1. Enter email
2. Backend returns list of societies for that email (names only, not codes)
3. If only 1 → auto-select, go to password screen
4. If multiple → show picker: "Which society?"
5. Enter password → logged in to selected society
```

Endpoint: `POST /api/v1/auth/lookup` — accepts email, returns society names the user belongs to.

## How to Test the Current Flow (via Swagger UI)

### Step 1 — Register a Society
- Open: `POST /api/v1/auth/society/register`
- Body:
```json
{
  "name": "Green Valley Society",
  "address": "123 Main Street, City",
  "contact_email": "admin@greenvalley.com"
}
```
- **Copy the `id` field** — this is your 5-letter society code (e.g. `MKQWZ`).

### Step 2 — Register the Admin User
- Open: `POST /api/v1/auth/register`
- Query param `society_id` = the 5-letter code you copied
- Body:
```json
{
  "full_name": "Your Name",
  "email": "you@example.com",
  "password": "yourpassword123"
}
```
- First user in a society auto-becomes ADMIN.

### Step 3 — Login
- Open: `POST /api/v1/auth/login`
- Query param `society_id` = the 5-letter code
- Body:
```json
{
  "email": "you@example.com",
  "password": "yourpassword123"
}
```
- Copy the `access_token` from the response.

### Step 4 — Authorise in Swagger
- Click the green **"Authorize"** button at the top
- Paste your token → click Authorize
- All protected endpoints now work

## Phase 5: Auth Redesign — Invite-Based Registration (Backend)
**Objective**: Replace public self-registration with admin-controlled invite flow.

### What changes:
1. **New: `InviteToken` model** (or fields on `User`):
   - `invite_token` (unique, short, URL-safe string)
   - `invite_expires_at` (e.g., 7 days from creation)
   - `is_activated` (boolean — false until resident sets password)
2. **Admin creates a member → invite token auto-generated**:
   - `POST /api/v1/members/` already exists — enhance it to return an invite token
   - Admin shares the token with the resident (out-of-band: SMS, WhatsApp, email)
3. **New endpoint: `POST /api/v1/auth/activate`**:
   - Resident sends `invite_token` + `password`
   - Backend validates token, sets hashed password, marks `is_activated = true`
   - Returns JWT — resident is immediately logged in
4. **Restrict or remove `POST /api/v1/auth/register`**:
   - Only admin-created invites should work (no open self-registration)
   - Keep `/auth/society/register` for society onboarding
5. **New endpoint: `POST /api/v1/auth/lookup`** (multi-society support):
   - Resident sends `email`
   - Backend returns list of `{society_id, society_name}` for all societies that email belongs to
   - Mobile app: if 1 result → auto-select; if multiple → show picker
   - Does NOT reveal passwords or tokens — only society names
6. **Login update: `POST /api/v1/auth/login`**:
   - Now accepts `society_id` + `email` + `password` (society_id comes from lookup step)
   - Admin login (web portal) still uses society code + email + password directly

### Migration:
- Add `invite_token`, `invite_expires_at`, `is_activated` columns to `users` table

## Phase 8.5: Bulk Excel Import — Units & Members
**Objective**: Allow admin to upload an Excel/CSV file to create hundreds of units and members in one go, instead of adding them one by one.

### Backend:
1. **New endpoint: `POST /api/v1/units/import`** (Admin-only):
   - Accepts an uploaded Excel (.xlsx) or CSV file
   - Parses rows into unit records (block_name, floor_number, unit_number, unit_type, area_sqft)
   - Validates all rows, returns errors for invalid rows (e.g. duplicates)
   - Creates all valid units in a single transaction
2. **New endpoint: `POST /api/v1/members/import`** (Admin-only):
   - Accepts an uploaded Excel/CSV file
   - Parses rows into member records (full_name, email, role, unit_number)
   - Auto-generates invite tokens for each member
   - Creates all members + sends back list of invite tokens
   - Validates: no duplicate emails within society, unit must exist
3. **Dependencies**: `openpyxl` (for .xlsx parsing), `python-multipart` (for file uploads)

### Admin Web Portal:
1. **Units page** — "Import from Excel" button → file picker → preview table → confirm → bulk create
2. **Members page** — "Import from Excel" button → file picker → preview table → confirm → shows invite tokens
3. **Template download** — provide downloadable Excel templates with correct column headers
